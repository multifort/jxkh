## 🔑 接口规范

### 基础信息

- **版本控制**: `/api/v1/...`
- **认证方式**: Bearer Token (JWT)
- **数据格式**: `application/json`
- **字符编码**: UTF-8

### 分页参数

```http
GET /api/plans?page=1&pageSize=20
```

### 统一响应格式

**成功响应**：
```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": 1712345678901
}
```

**失败响应**：
```json
{
  "code": 400,
  "message": "参数验证失败",
  "errors": [
    {
      "field": "name",
      "message": "名称不能为空"
    }
  ],
  "timestamp": 1712345678901
}
```

## 🔐 错误码说明

| 错误码 | 说明 | 处理建议 |
|--------|------|----------|
| 200 | 成功 | - |
| 201 | 创建成功 | - |
| 400 | 请求参数错误 | 检查请求参数 |
| 401 | 未认证或 Token 过期 | 重新登录 |
| 403 | 无权限访问 | 联系管理员 |
| 404 | 资源不存在 | 检查资源 ID |
| 409 | 资源冲突 | 检查数据唯一性 |
| 422 | 业务逻辑错误 | 查看错误详情 |
| 429 | 请求过于频繁 | 稍后重试 |
| 500 | 服务器内部错误 | 联系技术支持 |

## 🛠️ 开发建议

1. **前端开发**: 优先阅读 api-design.md，了解接口契约
2. **后端开发**: 根据 api-design.md 实现接口，保持文档同步
3. **测试人员**: 使用 Swagger UI 或 Postman 进行接口测试
4. **接口变更**: 先更新 api-design.md，再修改代码
