# 一、系统设计说明书（V1.0）

## 1. 项目背景与目标

### 1.1 项目背景

企业在绩效管理过程中普遍存在以下问题：

* 绩效指标分散，缺乏统一管理
* 绩效过程不可追踪（只看结果）
* 主观评价多，缺乏数据支撑
* 绩效与业务系统脱节
* 激励机制不透明

### 1.2 建设目标

构建一个一体化绩效管理平台，实现：

* 指标体系标准化（OKR / KPI / BSC）
* 绩效全过程闭环（制定 → 执行 → 评估 → 反馈）
* 数据驱动考核
* 多角色协同（员工 / 主管 / HR / 管理层）
* 支持组织规模扩展（集团化、多组织）

---

## 2. 系统总体架构

### 2.1 架构分层

```
表现层（Web / 移动端）
    ↓
业务层（绩效管理引擎）
    ↓
数据层（绩效数据 + 组织数据）
    ↓
集成层（ERP / OA / MES / HR系统）
```

### 2.2 技术架构建议

| 层级   | 技术                         |
| ---- | -------------------------- |
| 前端   | React / Vue3 + ECharts     |
| 后端   | Spring Boot / Node.js      |
| 数据库  | MySQL + Redis              |
| 数据分析 | ClickHouse / Elasticsearch |
| AI能力 | LLM（绩效分析 / 评价辅助）           |

---

## 3. 用户角色设计

| 角色   | 权限           |
| ---- | ------------ |
| 员工   | 查看目标、填报进度、自评 |
| 直属上级 | 制定目标、评分、反馈   |
| HR   | 规则配置、绩效周期管理  |
| 高管   | 绩效总览、组织分析    |
| 管理员  | 系统配置         |

---

## 4. 核心业务流程

### 4.1 绩效闭环流程

```
目标制定 → 目标确认 → 执行跟踪 → 数据采集 → 绩效评估 → 结果校准 → 反馈改进
```

### 4.2 关键节点

1. 绩效周期创建（季度/年度）
2. 指标分解（公司 → 部门 → 个人）
3. 执行过程跟踪（周/月）
4. 自动/手动评分
5. 绩效校准会议
6. 绩效结果发布

# 四、状态机设计（非常关键）

绩效系统的核心是“流程驱动”。

## 14. 绩效计划状态流转

```text
草稿
 ↓
待提交
 ↓
待审批
 ↓
执行中
 ↓
待评估
 ↓
已评估
 ↓
已校准
 ↓
已归档
```

---

## 15. 状态控制规则

| 当前状态 | 操作   | 下一个状态 |
| ---- | ---- | ----- |
| 草稿   | 提交   | 待审批   |
| 待审批  | 审批通过 | 执行中   |
| 执行中  | 周期结束 | 待评估   |
| 待评估  | 完成评分 | 已评估   |
| 已评估  | 校准   | 已校准   |

---

## 16. 状态机实现建议

后端建议：

* 使用 **状态枚举 + 状态流转表**
* 或引入状态机框架（如 Spring StateMachine）

# 七、评分计算引擎设计（核心能力）

## 28. 评分计算逻辑

```text
指标得分 × 权重 → 汇总 → 多维评分融合 → 最终得分
```

---

## 29. 示例计算

```text
KPI: 80 × 60% = 48
OKR: 90 × 30% = 27
360: 85 × 10% = 8.5

总分 = 83.5
```

---

## 30. 等级映射

| 分数    | 等级 |
| ----- | -- |
| 90+   | A  |
| 80-89 | B  |
| 70-79 | C  |
| <70   | D  |

---

## 31. 扩展能力（重要）

* 不同岗位不同评分模型
* 不同部门不同权重
* 动态调整公式（策略引擎）
