# 系统架构设计

## 1. 架构概述

### 1.1 架构目标
- **高可用性**：系统可用性 ≥ 99.9%
- **可扩展性**：支持用户规模从 100 到 10000+ 的平滑扩展
- **可维护性**：清晰的模块划分，便于迭代开发
- **安全性**：完善的权限控制和数据保护机制
- **性能**：核心接口响应时间 < 500ms

### 1.2 架构原则
- **分层架构**：表现层、业务层、数据层分离
- **单一职责**：每个模块只负责一个业务领域
- **依赖倒置**：高层模块不依赖低层模块细节
- **开闭原则**：对扩展开放，对修改关闭

## 4. 核心模块设计

### 4.1 用户认证模块

#### 认证流程
```
1. 用户登录 → 2. 验证用户名密码 → 3. 生成 JWT Token 
→ 4. 返回 Token → 5. 后续请求携带 Token → 6. 服务端验证 Token
```

#### 技术要点
- JWT 无状态认证
- Refresh Token 机制
- 密码 BCrypt 加密
- 登录失败次数限制

---

### 4.2 权限控制模块

#### RBAC 模型
```
用户(User) → 角色(Role) → 权限(Permission) → 资源(Resource)
```

#### 权限粒度
- **功能权限**：菜单、按钮级别
- **数据权限**：本人、本部门、全公司

#### 实现方式
- Spring Security + JWT
- 自定义注解 `@PreAuthorize`
- 数据权限拦截器

---

### 4.3 绩效管理引擎

#### 核心流程
```
目标制定 → 目标确认 → 执行跟踪 → 数据采集 → 绩效评估 
→ 结果校准 → 反馈改进 → 归档
```

#### 状态机设计
```java
enum PerformanceStatus {
    DRAFT,          // 草稿
    PENDING_SUBMIT, // 待提交
    PENDING_APPROVE,// 待审批
    IN_PROGRESS,    // 执行中
    PENDING_EVAL,   // 待评估
    EVALUATED,      // 已评估
    CALIBRATED,     // 已校准
    ARCHIVED        // 已归档
}
```

#### 评分计算引擎
```
最终得分 = Σ(指标得分 × 权重) × 评价维度权重
```

支持动态配置：
- KPI 权重 60%
- OKR 权重 30%
- 360 评价权重 10%
